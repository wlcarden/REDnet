# Onboarding PWA — prototype

The one bespoke, security-sensitive component of REDnet (`SPEC §5 Track A`): a `matrix-js-sdk` PWA that creates a no-PII account and bootstraps E2EE **silently** — the heart of the ~2-tap onboarding.

## Milestone A — silent E2EE bootstrap ✅ PASS (2026-06-16, Synapse 1.155.0, matrix-js-sdk 41.8.0-rc.0)

**Question:** can we create a fully encrypted-capable account (cross-signing + key backup) with **no user-visible recovery key and no UIA prompts**? The research said yes (MSC3967 + an app-supplied recovery key); this proves it end-to-end.

**Run:** `bash run.sh` (stands up Synapse, registers a user, runs `onboard.mjs`, writes `result.json`).

**Result:**

```json
{
  "crossSigning_publicKeysOnDevice": true,
  "crossSigning_privateKeysInSecretStorage": true,
  "keyBackup_activeVersion": "2",
  "keyBackup_onServer": "2",
  "recoveryKey_generatedByApp": true,
  "recoveryKey_lengthChars": 59,
  "recoveryKey_shownToUser": false,
  "errors": [],
  "PASS": true
}
```

**Confirmed:**

- ✅ **Silent cross-signing** — first `keys/device_signing/upload` returns **200 on empty auth** (MSC3967: no UIA when no master key exists). Validated empirically.
- ✅ **Silent secret storage + key backup** — a recovery key is generated by the app, used to seed 4S and the key backup, and **never displayed**. The user gets a fully E2EE account with zero prompts.
- ✅ The recovery key (`encodedPrivateKey`, 59 chars) is held in app memory for the session — `accept-loss` means we discard it (no escrow); the v2 hooks would custody a derived claim instead.

## Implementation gotchas (real, for whoever builds the production PWA)

1. **You must `startClient()` (start the sync loop) before bootstrapping.** Without it, the bootstrap awaits hang on background `/keys/query` and (in Node) the process exits 0 on an empty event loop with nothing done. Element does login → `initRustCrypto` → `startClient` → bootstrap; replicate that order.
2. **Supply `cryptoCallbacks.getSecretStorageKey`** (+ `cacheSecretStorageKey`) returning the app-held key. Without it, exporting cross-signing keys into 4S throws `No getSecretStorageKey callback supplied`. This callback is _how_ the recovery key stays invisible — the app hands the key back to the SDK instead of asking the user.
3. **Silent bootstrap only works on a _fresh_ account** (no existing cross-signing master key — MSC3967). Resetting existing cross-signing requires UIA. So this runs exactly once, at first onboarding — which is the design.
4. (Node-only) `process.exit()` truncates piped stdout — write results to a file.

## Milestone D — Element Web hand-off / re-nag ✅ characterized (`handoff.mjs`, `run-handoff.sh`)

**Question (original Spike 02):** after the silent bootstrap, does Element Web re-nag for recovery? **Answer: depends entirely on whether it's the SAME device.** Proven headlessly (the device/verification state is exactly what drives Element Web's nag):

```
device1 (the bootstrapped onboarding session): crossSigningVerified=true,  keysCached=true  → NO nag
device2 (a FRESH Element Web login):           crossSigningVerified=false, keysCached=false,
                                               secretStorage set up on server=true          → WOULD re-nag
```

**Conclusion (now corrected in `SPEC §5`):** the silent crypto bootstrap must run on **Element Web's own session/device** — i.e. a **customized Element Web first-run**, not a separate PWA that hands a token to a fresh Element Web login. A fresh login is an unverified new device that can't unlock the already-set-up secret storage, so Element Web prompts "verify this session / enter your recovery key." Same-session = no nag.

## Milestone E — Phase-1 self-held passphrase RECOVERY ✅ PASS (`phase1-recovery.mjs`, `run-phase1.sh`)

**Question:** can a member who loses their device recover their **identity** and **message history** on a
brand-new device using ONLY a passphrase — no old device, no moderators (the bootstrap recovery model,
works at community size N=1)? This is native Matrix 4S keyed by a passphrase; the soft fork drives it.

**Run:** `bash run-phase1.sh` (two devices, passphrase-only recovery; writes `result-phase1.json`).

**Result:**

```json
{
  "device2_derivedKeyFromPassphrase": true,
  "device2_crossSigning_privateKeysCached": true,
  "device2_recoveredSameIdentity": true,
  "device2_keyBackup_imported": 1,
  "device2_keyBackup_total": 1,
  "wrongPassphrase_yieldsDifferentKey": true,
  "PASS": true
}
```

**Confirmed:** a fresh device 2 re-derived the 4S key from the passphrase, recovered the **same**
cross-signing master key (recovered, not reset), and **restored a message backed up before device 2
existed** — all from the passphrase alone. A wrong passphrase yields a different key (it truly gates).
Verified logic ported into the CryptoSetupExtensions module at
`deploy/element-web/rednet-module/src/onboarding.ts` (`silentBootstrap` / `recoverWithPassphrase`).

**Gotchas this flushed out** (both about sequencing against background machinery, not recovery logic):

1. **`getSecretStorageKey` callback race** — it fires before `createSecretStorageKey` resolves; pre-generate the key.
2. **Plaintext-send footgun** — `sendEvent` right after `createRoom` races the `m.room.encryption` state, so the message goes out unencrypted → no megolm key → nothing to back up. Wait for `isEncryptionEnabledInRoom` first. (Backup was working; it had nothing to do.)
